# DESIGN.md — ax-builder UI/UX 사양

> UI/UX는 이 문서가 SSoT. PRD는 **무엇을**, 이 문서는 **어떻게 보여주고 어떻게 반응하는가**.
> PRD §3~§11의 범위 내에서 화면·상태별 레이아웃·인터랙션·카피를 확정한다.

---

## 1. 원칙

1. **비개발자 우선 언어** — "API 키가 잘못되었습니다"가 아니라 "입력하신 키가 거부됐어요. 다시 확인해주세요."
2. **상태는 항상 보이게** — 지금 Planning인지 Building인지 QA인지 awaiting_env인지, 클릭 없이 한눈에.
3. **실패는 분류로 말한다** — FailureClassifier(ADR 0002) 결과를 그대로 UI 메시지로.
4. **유저가 고칠 수 있는 것과 없는 것을 분리** — "[재시도]"와 "[다시 입력]"과 "AI가 수정 중..."은 다른 버튼.
5. **system-injected env는 절대 보여주지 않는다** — `AX_AI_TOKEN` 같은 건 유저의 멘탈 모델 밖.

## 2. 프로젝트 상태별 화면 매핑

PRD §10.1 상태 머신 대응 UI:

| 상태 | 카드/화면 | 주 CTA |
|---|---|---|
| `draft` | 채팅 화면 (빈 메시지 placeholder) | "어떤 걸 만들고 싶으세요?" |
| `planning` | 채팅 + 완성도 사이드바 | 없음 (AI 대화) |
| `plan_ready` | 채팅 + "빌드 시작" 버튼 활성 | **[빌드 시작]** |
| `building` | 빌드 진행 화면 (phase 리스트) | [빌드 중단] |
| `qa` | 빌드 진행 화면 + "검증 중" 배지 | — |
| `awaiting_env` | ENV 입력 화면 (§4) | **[적용하기]** |
| `env_qa` | 빌드 진행 화면 + "환경 검증 중" | [재시도] (transient 시) |
| `deployed` | 프로젝트 카드에 URL + "수정 요청" | **[열기]** / [수정 요청] |
| `failed` | 실패 카드 + 원인 요약 + "다시 시도" | [PRD 수정하기] 또는 [재시도] |
| `modifying` | 채팅 + "수정 중" 배지 | 없음 |

### 3.3 실시간 로그 (PRD §4.10 agent_logs 기반)

- 상시 표시되는 **요약 라인**은 자연어: `"App.tsx 작성 중"`, `"의존성 설치 중"`, `"서버가 어느 포트에 떴는지 확인 중"`.
- "세부 내용 보기" 토글 시 **원문 stdout** + **tool_use/tool_result** 스트림 표시.
- 색상: assistant = 흰색, tool_call = 파란색, tool_result = 녹색, error = 빨간색.

## 4. ENV 입력 화면 (2-모드, ADR 0005 + 0006)

같은 `EnvInput.tsx` 컴포넌트가 프로젝트 상태에 따라 두 모드로 렌더링된다.

### 4.1 유지보수 모드 (mode = maintenance)

`project.state === 'deployed'`에서 진입. 필드별 독립 편집이 기본.

```
┌──────────────────────────────────────────────────────────┐
│  ← 대시보드   환경 설정               🟢 운영 중 (mock)    │
│                                                            │
│  필요할 때 값을 넣으면 실제 기능이 활성화돼요.             │
│  지금은 ⚠ mock 응답으로 동작 중이에요.                     │
│                                                            │
│  ━━ 필수 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━   │
│                                                            │
│  🔑 STRIPE_SECRET_KEY                                      │
│     결제 기능을 위해 Stripe 키가 필요해요.                 │
│     ⓘ 발급 방법: Stripe 대시보드 → 개발자 → API 키        │
│     [••••••••5432                              ] [보기▾]  │
│     패턴 OK · 길이 OK                                      │
│                                                            │
│  ━━ 선택 (펼치기 ▸) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━   │
│                                                            │
│              [💾 저장]        [💾 저장 후 재시작]          │
└──────────────────────────────────────────────────────────┘
```

- **[💾 저장]**: PUT `/env { apply:false }` — DB만 갱신, 컨테이너 영향 無. 토스트 "저장됨".
- **[💾 저장 후 재시작]**: PUT `/env { apply:true }` — DB 갱신 + docker restart + 헬스체크. 진행 중엔 버튼 disabled + spinner.

### 4.2 초기 설정 모드 (mode = setup, 레거시)

`project.state === 'awaiting_env'`에서 진입. ADR 0005 이후에는 거의 사용되지 않지만 호환 유지.

```
┌──────────────────────────────────────────────────────────┐
│  ← 대시보드   환경 설정                                    │
│                                                            │
│  이 앱을 실행하려면 아래 값이 필요해요.                    │
│                                                            │
│  ━━ 필수 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━   │
│                                                            │
│  🔑 STRIPE_SECRET_KEY *                                    │
│     [sk_test_xxxxxxxxxxxx___________________]              │
│                                                            │
│                                    [적용하기]              │
└──────────────────────────────────────────────────────────┘
```

원샷 제출 → status `env_qa`로 전환.

### 4.3 인라인 밸리데이션

`.env.example` 메타라인에서 추출한 `validation_pattern` / `min_length` / `max_length` 기준. 실시간 표시:

- ✅ 녹색 "패턴 OK · 길이 OK"
- ⚠ 빨간 "패턴 불일치 (예: sk_test_...)" 또는 "길이 부족 (10자 이상)"
- 서버 400 응답 오면 동일 필드에 에러 매핑.

패턴이 없는 변수는 형식 체크 없음 — Claude Code가 모르는 서비스에 과잉 제약 걸지 않도록.

### 4.4 보안 UX

- 값은 입력 직후 마스킹 (`sk_test_••••••••5432`). [보기▾] 토글로 평문 확인 (재렌더 시 자동 마스킹 복귀).
- 서버 응답에도 마스킹된 preview만. 원문은 저장 시점에만 서버에 전달.
- `system-injected` 변수는 **렌더링하지 않음** — 유저의 멘탈 모델에 존재하지 않는다.
- 입력 필드는 `autocomplete="off"` + `type="password"` 기본.

## 5. 프로젝트 대시보드

### 5.1 카드 상태 배지

| 상태 | 배지 색 | 배지 문구 | 보조 라인 |
|---|---|---|---|
| planning | 회색 | 기획 중 | "AI와 대화 중" |
| plan_ready | 하늘 | 빌드 대기 | "빌드 시작 가능" |
| building | 파랑 | 빌드 중 | "phase 3/7" |
| qa | 보라 | 검증 중 | "앱이 정상 기동하는지 확인 중" |
| awaiting_env | 주황 | 설정 필요 | "API 키를 입력해주세요" (레거시) |
| env_qa | 보라 | 환경 검증 중 | "재시작해서 다시 기동 중" |
| deployed (mock) | 녹색 + ⚠ | 운영 중 (mock) | URL + "일부 기능은 환경 설정 후 활성화" |
| deployed (real) | 녹색 | 운영 중 | URL 표시 |
| failed | 빨강 | 실패 | 분류 결과 요약 |
| modifying | 노랑 | 수정 중 | "업데이트 중" |

**mock 상태 구분**: `project_env_vars` 중 `tier='user-required'` AND `value_ciphertext IS NULL`인 행이 1개 이상이면 "mock"으로 표시. 전부 채워지면 "real".

### 5.1.1 deployed 카드 버튼 (ADR 0006)

```
┌───────────────────────────────────────────┐
│ 🎮 라이어 게임       🟢 운영 중 (mock) ⚠    │
│                                             │
│ 🌐 https://app-abc.local                    │
│ 일부 기능은 환경 설정 후 활성화됩니다.       │
│                                             │
│ [🎮 열기] [⚙ 환경 설정] [🔄 재시작] [수정 요청] │
└───────────────────────────────────────────┘
```

| 버튼 | 권한 | 동작 |
|---|---|---|
| 🎮 열기 | 누구나 | 앱 URL 새 탭 |
| ⚙ 환경 설정 | owner, editor | `/projects/:id/env` (maintenance 모드) |
| 🔄 재시작 | **owner만** | 확인 모달 → POST `/restart` |
| 수정 요청 | owner, editor | 채팅 세션(modifying) 생성 |

## 6. 실패 상세 패널 (개발자용)

"세부 내용 보기"를 펼쳤을 때:

```
분류: code_bug
근거: regex 매칭 "TypeError: Cannot read properties of undefined"

에러 스택 (마지막 30줄):
  TypeError: Cannot read properties of undefined (reading 'filter')
    at getAvailableRooms (server/roomManager.js:42)
    ...

관련 로그: [전체 보기]
```

- "전체 보기"는 `build_phases.output_log` + `agent_logs` 전문 뷰어로 이동.
- 비개발자는 보지 않는 영역 — 기본 접힘.

## 7. 마이크로카피 톤

- 성공: "만들었어요!", "완료됐어요"
- 진행: "~하고 있어요", "~ 중..."
- 실패(유저 귀책): "~을 확인해주세요", "다시 입력해주세요"
- 실패(일시): "잠시 뒤 다시 시도해요"
- 실패(AI 귀책): "기획을 다시 다듬어야 해요"
- 금지어: "오류", "에러", "실패" (개발자용 패널에서만 사용)

## 8. 상태 전이 애니메이션

- phase 완료 시 체크 아이콘 페이드인 (200ms)
- `building → qa` 전환 시 상단 배너가 "빌드 중 → 검증 중"으로 slide
- `awaiting_env` 진입 시 화면 전환은 없이 **인라인 모달**로 ENV 입력 유도 (컨텍스트 이탈 최소화)
- 실패 상태는 shake 애니메이션 없음 (UX 자극 방지)

---

## 9. 반응형 / 접근성

- 모바일: 카드 full-width, 버튼 44px 높이 이상.
- 키보드: 모든 CTA Tab 접근. ENV 입력은 `Enter`로 다음 필드.
- 스크린 리더: 상태 배지에 `aria-label="상태: 설정 필요"`.
- 로그 스트림은 `aria-live="polite"`로 설정 (과도한 낭독 방지).

## 10. 재시작 확인 모달 (ADR 0006)

```
┌──────────────────────────────────────┐
│  🔄 프로젝트 재시작                   │
│                                        │
│  정말 재시작하시겠습니까?              │
│  약 5~10초 동안 서비스가 잠시 끊겨요.  │
│                                        │
│  [취소]                    [재시작하기]│
└──────────────────────────────────────┘
```

- trigger: `deployed` 카드의 🔄 버튼 (owner 전용)
- 확인 후 POST `/restart` — 프로그레스 인디케이터 대체 표시, 10초 헬스체크 응답 기다림
- 응답별 토스트:
  - 성공: "재시작 완료"
  - transient/env_rejected: "아직 응답이 없어요, 값 확인 후 다시 시도" (deployed 유지)
  - code_bug: "통합 로직에 문제가 있어요. 수정 요청으로 넘어갈까요?" → modifying 세션 생성 유도
