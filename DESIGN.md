# DESIGN.md — ax-builder UI/UX 사양

> UI/UX는 이 문서가 SSoT. PRD는 **무엇을**, 이 문서는 **어떻게 보여주고 어떻게 반응하는가**.
> PRD §3~§11의 범위 내에서 화면·상태별 레이아웃·인터랙션·카피를 확정한다.

---

## 1. 원칙

1. **비개발자 우선 언어** — "API 키가 잘못되었습니다"가 아니라 "입력하신 키가 거부됐어요. 다시 확인해주세요."
2. **상태는 항상 보이게** — 지금 Planning인지 Building인지 QA인지 awaiting_env인지, 클릭 없이 한눈에.
3. **실패는 분류로 말한다** — FailureClassifier(ADR 0002) 결과를 그대로 UI 메시지로.
4. **유저가 고칠 수 있는 것과 없는 것을 분리** — "[재시도]"와 "[다시 입력]"과 "AI가 수정 중..."은 다른 버튼.
5. **system-injected env는 절대 보여주지 않는다** — `AX_AI_TOKEN` 같은 건 유저의 멘탈 모델 밖.
6. **도구 결과는 UI가 직접 표시한다** — AI 텍스트 응답은 보조. 예를 들어 `propose_handoff` 결과는 오케스트레이터가 WS 이벤트(`phase=plan_ready` / `update_ready` / `handoff_rejected`)로 배너를 띄우고, AI가 채팅에서 뭐라고 요약하든 그 배너가 진실. (회고 §5 — AI 환각 "이관 완료" 사례 대응.)

| `failed` | 실패 카드 + 원인 요약 + 분기별 CTA | [PRD 수정하기] / [재시도] / [관리자 문의] |

## 3. 빌드 진행 화면 (BuildStatus)

### 3.1 레이아웃 (정상)

```
┌──────────────────────────────────────────────────────────┐
│  ← 대시보드   빌드 진행 상태  🟢 실시간      경과: 02:14   │
│                                                            │
│  frontend_ui                    67%    [■ 빌드 중단]        │
│  ████████████████░░░░░░░░                                  │
│                                                            │
│  ✅ scaffold                      완료                     │
│  ✅ backend_core                  완료                     │
│  🔄 frontend_ui                   진행 중...               │
│     └─ "App.tsx 작성 중"                                   │
│  ⬜ integration                                            │
│  ⬜ qa                                                     │
│                                                            │
│  ┌─ 실시간 로그 ─────────────────────────────────────┐     │
│  │ > Writing src/App.tsx...                        │     │
│  │ > Installing dependencies...                    │     │
│  └─────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────┘
```

진행 바 우측에 **"■ 빌드 중단"** 버튼. 업데이트 라인이면 **"↺ 업데이트 중단"**으로 카피·진행바 시안색. 클릭 시 확인 모달 → `POST /build/cancel`. 업데이트 라인 중단 시 "이전 버전은 그대로 유지됩니다" 문구 포함.

### 3.1.1 레이아웃 (실패 — retry-first 정책)

첫 실패 또는 1회 실패 상태 (primary = 재시도):

```
┌──────────────────────────────────────────────────────────┐
│  ← 대시보드   빌드 진행 상태                 경과: 05:12   │
│                                                            │
│  ┌─ 🔴 빌드 실패 ──────────────────────────────────────┐   │
│  │ Building Agent가 비정상 종료했습니다 (exit 137)     │   │
│  │                                                    │   │
│  │ 프로세스가 중단됐거나 실행 환경 문제였다면         │   │
│  │ "다시 빌드"로 재시도할 수 있어요. 기획 자체에       │   │
│  │ 수정이 필요하면 "기획 대화로"를 선택하세요.        │   │
│  │                                                    │   │
│  │              [■ ↻ 다시 빌드]  [기획 대화로]          │   │
│  └────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────┘
```

**2회 이상 연속 실패 시 (primary 플립 — "기획 대화로"가 기본값)**:

```
┌──────────────────────────────────────────────────────────┐
│  ┌─ 🔴 빌드 실패 (연속 2회) ─────────────────────────┐    │
│  │                                                    │   │
│  │ 같은 기획으로 여러 번 실패했어요.                   │   │
│  │ 대화로 보강한 뒤 다시 빌드하는 편이 효과적입니다.   │   │
│  │                                                    │   │
│  │              [■ 기획 대화로]  [↻ 다시 빌드]          │   │
│  └────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────┘
```

정책 (ARCHITECTURE §7.6 retry-first):
- **"↻ 다시 빌드"** = `POST /build/retry` → `failed → plan_ready → building`. 같은 handoff로 재시작.
- **연속 실패 카운터**는 `builds.getConsecutiveFailureCount()` — 가장 최근 성공 이후의 실패(failed/bounced/cancelled) 수. 2 이상이면 CTA 순서 뒤집기.
- Claude Code 실행이 확률적이라 1차 실패는 대부분 재시도로 풀림. 2차부터는 진짜 기획 이슈 신호로 간주.

### 3.3 실시간 로그 (PRD §4.10 agent_logs 기반)

- 상시 표시되는 **요약 라인**은 자연어: `"App.tsx 작성 중"`, `"의존성 설치 중"`, `"서버가 어느 포트에 떴는지 확인 중"`.
- "세부 내용 보기" 토글 시 **원문 stdout** + **tool_use/tool_result** 스트림 표시.
- 색상: assistant = 흰색, tool_call = 파란색, tool_result = 녹색, error = 빨간색.

## 4. ENV 입력 화면 (2-모드, ADR 0005 + 0006)

같은 `EnvInput.tsx` 컴포넌트가 프로젝트 상태에 따라 두 모드로 렌더링된다.

### 4.1 유지보수 모드 (mode = maintenance)

`project.state === 'deployed'`에서 진입. 필드별 독립 편집이 기본.

```
│                                                            │
│  ━━ 필수 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━   │

│     결제 기능을 위해 Stripe 키가 필요해요.                 │
│     ⓘ 발급 방법: Stripe 대시보드 → 개발자 → API 키        │
│     [••••••••5432                              ] [보기▾]  │

│                                                            │
│  ━━ 선택 (펼치기 ▸) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━   │
```

- **[💾 저장]**: PUT `/env { apply:false }` — DB만 갱신, 컨테이너 영향 無. 토스트 "저장됨".
- **[💾 저장 후 재시작]**: PUT `/env { apply:true }` — DB 갱신 + docker restart + 헬스체크. 진행 중엔 버튼 disabled + spinner.

### 4.2 초기 설정 모드 (mode = setup, 레거시)

`project.state === 'awaiting_env'`에서 진입. ADR 0005 이후에는 거의 사용되지 않지만 호환 유지.

```
┌──────────────────────────────────────────────────────────┐
│  ← 대시보드   환경 설정                                    │
│                                                            │
│  이 앱을 실행하려면 아래 값이 필요해요.                    │
│                                                            │
│  ━━ 필수 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━   │
│                                                            │
│  🔑 STRIPE_SECRET_KEY *                                    │
│     [sk_test_xxxxxxxxxxxx___________________]              │
│                                                            │
│                                    [적용하기]              │
└──────────────────────────────────────────────────────────┘
```

원샷 제출 → status `env_qa`로 전환.

### 4.3 인라인 밸리데이션

`.env.example` 메타라인에서 추출한 `validation_pattern` / `min_length` / `max_length` 기준. 실시간 표시:

- ✅ 녹색 "패턴 OK · 길이 OK"
- ⚠ 빨간 "패턴 불일치 (예: sk_test_...)" 또는 "길이 부족 (10자 이상)"
- 서버 400 응답 오면 동일 필드에 에러 매핑.

패턴이 없는 변수는 형식 체크 없음 — Claude Code가 모르는 서비스에 과잉 제약 걸지 않도록.

### 4.4 보안 UX

- 값은 입력 직후 마스킹 (`sk_test_••••••••5432`). [보기▾] 토글로 평문 확인 (재렌더 시 자동 마스킹 복귀).
- 서버 응답에도 마스킹된 preview만. 원문은 저장 시점에만 서버에 전달.
- `system-injected` 변수는 **렌더링하지 않음** — 유저의 멘탈 모델에 존재하지 않는다.
- 입력 필드는 `autocomplete="off"` + `type="password"` 기본.

## 5. 프로젝트 대시보드

### 5.1 카드 상태 배지

**첫 빌드 라인**
| 상태 | 배지 색 | 배지 문구 | 보조 라인 |
|---|---|---|---|
| planning | 회색 | 기획 중 | "AI와 대화 중" |
| plan_ready | 하늘 | 빌드 대기 | "빌드 시작 가능" |
| building | 파랑 | 빌드 중 | "phase 3/7" |
| qa | 보라 | 검증 중 | "앱이 정상 기동하는지 확인 중" |

**env 사이드**
| 상태 | 배지 색 | 배지 문구 | 보조 라인 |
|---|---|---|---|
| awaiting_env | 주황 | 설정 필요 | "API 키를 입력해주세요" (레거시) |
| env_qa | 보라 | 환경 검증 중 | "재시작해서 다시 기동 중" |

**업데이트 라인** (ADR 0008 — 첫 빌드와 색·카피 달리해서 맥락 구분)
| 상태 | 배지 색 | 배지 문구 | 보조 라인 |
|---|---|---|---|
| planning_update | 인디고 | 수정 기획 중 | "추가·수정 내용을 AI와 논의 중" |
| update_ready | 민트 | 업데이트 대기 | "업데이트 시작 가능" |
| updating | 시안 | 업데이트 중 | "변경 사항 적용 중 (기존 앱 계속 운영)" |
| update_qa | 시안 | 회귀 검증 중 | "기존 기능·새 기능 모두 확인 중" |

**터미널**
| 상태 | 배지 색 | 배지 문구 | 보조 라인 |
|---|---|---|---|
| deployed (mock) | 녹색 + ⚠ | 운영 중 (mock) | URL + "일부 기능은 환경 설정 후 활성화" |
| deployed (real) | 녹색 | 운영 중 | URL 표시 |
| failed | 빨강 | 실패 | 분류 결과 요약 |

**색 체계 원칙**: 첫 빌드(회색·하늘·파랑·보라) vs 업데이트(인디고·민트·시안)는 한눈에 다른 **페이즈**임을 암시. 유저가 "지금 이건 새로 만드는 중인지, 기존 앱 건드리는 중인지"를 색 하나로 인지.

**mock 상태 구분**: `project_env_vars` 중 `tier='user-required'` AND `value_ciphertext IS NULL`인 행이 1개 이상이면 "mock"으로 표시. 전부 채워지면 "real".

**업데이트 중 배너**: `updating` / `update_qa` 상태의 프로젝트 카드엔 반드시 **"기존 앱은 계속 접속 가능합니다"** 문구 + 기존 배포 URL 유지. 유저 안심 UX.

### 5.1.2 Chat 사이드바 readiness 배지

`planning` / `planning_update` 상태에서 채팅 화면 사이드바 상단에 스코어와 상태 라벨 노출. `propose_handoff`의 두 조건과 정합되게 3단계로 구분 (원칙 #6 — 도구 결과와 UI 표시가 어긋나지 않음):

| 조건 | 스코어바 색 | 라벨 | 의미 |
|---|---|---|---|
| `min_completeness >= 0.85` + unresolved 0 | 초록 | 충분 조건 충족 | `propose_handoff` accepted 확실 |
| `0.6 <= min < 0.85` + unresolved 0 | 노랑 | 최소 조건 충족 (보강 권장) | `propose_handoff`가 "보강 권장"으로 거부될 수 있음 — AI가 자체 평가에서 `is_sufficient=false`를 낼 경우 |
| 그 외 | 빨강 | 보강 필요 | 빌드 시작 불가 |

노랑 상태에선 추가 경고문: "⚠ 최소 조건만 충족 — propose_handoff가 '보강 권장'으로 거부될 수 있어요."

이 구분 없이 "빌드 가능"만 표시하던 이전 UI는 유저 혼란의 원인이 됐음 — 사이드바는 초록인데 propose_handoff는 거부되는 모순 (스크린샷 회귀 기록).

```
│                                             │
│ 🌐 localhost:3001    [접속 →]               │

│                                             │
│ [⚙ 환경 설정] [🔄 재시작] [수정 요청] [삭제]  │
└───────────────────────────────────────────┘
```

접속은 URL 우측 인라인 링크(`접속 →`)로 제공 — 별도 "열기" 버튼이 버튼 라인을 차지하지 않아 관리 CTA를 더 눈에 띄게 한다.

| 요소 | 권한 | 동작 |
|---|---|---|
| 🌐 URL + 접속 → | 누구나 | 앱 URL 새 탭 (a 태그, `target=_blank`) |
| ⚙ 환경 설정 | owner, editor | `/projects/:id/env` (maintenance 모드) |
| 🔄 재시작 | **owner만** | 확인 모달 → POST `/restart` |
| 수정 요청 | owner, editor | 채팅 진입. 첫 메시지 시 `deployed → planning_update` 전이 (ADR 0008) |
| 삭제 | **owner만** | 확인 모달 → DELETE `/projects/:id` |

## 8. 상태 전이 애니메이션

- phase 완료 시 체크 아이콘 페이드인 (200ms)
- `building → qa` 전환 시 상단 배너가 "빌드 중 → 검증 중"으로 slide
- `awaiting_env` 진입 시 화면 전환은 없이 **인라인 모달**로 ENV 입력 유도 (컨텍스트 이탈 최소화)
- 실패 상태는 shake 애니메이션 없음 (UX 자극 방지)

## 9. 반응형 / 접근성

- 모바일: 카드 full-width, 버튼 44px 높이 이상.
- 키보드: 모든 CTA Tab 접근. ENV 입력은 `Enter`로 다음 필드.

## 10. 재시작 확인 모달 (ADR 0006)

```
┌──────────────────────────────────────┐
│  🔄 프로젝트 재시작                   │
│                                        │
│  정말 재시작하시겠습니까?              │
│  약 5~10초 동안 서비스가 잠시 끊겨요.  │
│                                        │
│  [취소]                    [재시작하기]│
└──────────────────────────────────────┘
```

- trigger: `deployed` 카드의 🔄 버튼 (owner 전용)
- 확인 후 POST `/restart` — 프로그레스 인디케이터 대체 표시, 10초 헬스체크 응답 기다림
- 응답별 토스트:
  - 성공: "재시작 완료"
  - transient/env_rejected: "아직 응답이 없어요, 값 확인 후 다시 시도" (deployed 유지)
  - code_bug: "통합 로직에 문제가 있어요. 수정 요청으로 넘어갈까요?" → modifying 세션 생성 유도
